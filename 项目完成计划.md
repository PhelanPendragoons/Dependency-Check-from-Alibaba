## 一、项目全景概览

### 1.1 项目是做什么的？

集团内部有数万个Java、前端等应用，这些应用依赖了大量的第三方组件（比如Spring框架、Log4j日志库等）。这些第三方组件可能存在**安全漏洞**（比如Log4j的"核弹级"漏洞）或**许可证问题**（比如用了GPL协议的库可能导致商业纠纷）。

本项目就是要做一个**内部工具平台**，让开发人员上传自己的项目代码，系统自动扫描出所有依赖的安全漏洞和许可证问题，并生成报告。

#### 1.2 配置Maven阿里云镜像

##### 为什么要配镜像？

Maven默认从 **Maven Central（中央仓库）** 下载项目依赖的JAR包，但这个仓库的服务器在国外。国内下载速度非常慢（可能只有几十KB/s），而且经常超时失败。

阿里云在国内提供了Maven镜像服务，速度可以达到几MB/s。**配置镜像后，Maven会从阿里云下载依赖，而不是从中央仓库。**

##### 第一步：找到settings.xml文件

Maven的配置文件叫 `settings.xml`，有两个位置：

| 位置 | 路径（Windows） | 说明 |
|------|----------------|------|
| **用户级别（推荐）** | `C:\Users\你的用户名\.m2\settings.xml` | 只对当前用户生效，推荐在这里配置 |
| 全局级别 | `D:\apache-maven-3.8.x\conf\settings.xml` | 对所有用户生效，需要管理员权限 |

> **注意**：`.m2` 文件夹默认是隐藏的，需要在文件资源管理器中开启"查看隐藏的项目"。

##### 第二步：检查文件是否存在

打开文件资源管理器，导航到 `C:\Users\你的用户名\.m2\` 目录：

- 如果已经有 `settings.xml` 文件 → 直接用记事本/VS Code打开编辑
- 如果没有 `settings.xml` 文件 → 需要手动创建一个

##### 第三步：创建/编辑settings.xml

用VS Code或记事本打开 `settings.xml`，将以下内容**完整复制**进去：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<settings xmlns="http://maven.apache.org/SETTINGS/1.0.0"
          xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
          xsi:schemaLocation="http://maven.apache.org/SETTINGS/1.0.0
                              http://maven.apache.org/xsd/settings-1.0.0.xsd">

    <!-- 配置阿里云镜像 -->
    <mirrors>
        <mirror>
            <id>aliyunmaven</id>
            <name>阿里云公共仓库</name>
            <!-- mirrorOf * 表示所有依赖都从阿里云下载 -->
            <mirrorOf>*</mirrorOf>
            <url>https://maven.aliyun.com/repository/public</url>
        </mirror>
    </mirrors>

    <!-- （可选）配置JDK版本，避免编译时提示"不再支持源选项 5" -->
    <profiles>
        <profile>
            <id>jdk-17</id>
            <activation>
                <activeByDefault>true</activeByDefault>
                <jdk>17</jdk>
            </activation>
            <properties>
                <maven.compiler.source>17</maven.compiler.source>
                <maven.compiler.target>17</maven.compiler.target>
                <maven.compiler.compilerVersion>17</maven.compiler.compilerVersion>
            </properties>
        </profile>
    </profiles>

</settings>
```

##### 第四步：验证配置是否生效

打开命令行（CMD或PowerShell），执行：

```bash
# 查看Maven版本和settings.xml位置
mvn -version
```

如果配置成功，输出中会显示：
```
...
User settings file: C:\Users\你的用户名\.m2\settings.xml
...
```

然后尝试编译一个项目，观察下载速度：

```bash
# 进入项目目录后执行
mvn clean install -DskipTests
```

如果看到下载速度达到 **1MB/s以上**，说明配置成功！

##### 第五步：如果下载仍然很慢怎么办？

如果配置后下载速度仍然不理想，可以尝试以下方案：

**方案一：使用阿里云其他镜像仓库**

```xml
<mirrors>
    <!-- 阿里云中央仓库镜像 -->
    <mirror>
        <id>aliyun-central</id>
        <mirrorOf>central</mirrorOf>
        <name>阿里云中央仓库镜像</name>
        <url>https://maven.aliyun.com/repository/central</url>
    </mirror>
    <!-- 阿里云Spring插件镜像 -->
    <mirror>
        <id>aliyun-spring</id>
        <mirrorOf>spring</mirrorOf>
        <name>阿里云Spring仓库镜像</name>
        <url>https://maven.aliyun.com/repository/spring</url>
    </mirror>
</mirrors>
```

**方案二：在项目pom.xml中直接配置仓库（不依赖settings.xml）**

```xml
<repositories>
    <repository>
        <id>aliyun</id>
        <name>阿里云仓库</name>
        <url>https://maven.aliyun.com/repository/public</url>
    </repository>
</repositories>
```

**方案三：使用华为云镜像（阿里云镜像的备选）**

```xml
<mirror>
    <id>huaweicloud</id>
    <mirrorOf>*</mirrorOf>
    <name>华为云镜像</name>
    <url>https://repo.huaweicloud.com/repository/maven/</url>
</mirror>
```

##### 常见问题

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| `Could not transfer artifact` | 网络连接失败 | 检查网络，或切换为华为云镜像 |
| `settings.xml` 报错 | XML格式错误 | 检查标签是否闭合，不要有中文标点 |
| 配置后仍然从中央仓库下载 | settings.xml位置不对 | 确认文件在 `C:\Users\用户名\.m2\` 目录下 |
| 下载速度还是很慢 | 可能是公司网络限制 | 尝试使用VPN，或联系IT部门开放Maven仓库访问 |

###### ❌ 情况一：提示 `'node' 不是内部或外部命令`

**原因**：Node.js没有安装，或者安装时没有勾选"Add to PATH"（添加到环境变量）。

**解决方法（按顺序尝试）：**

**方法A：检查Node.js是否已安装（推荐新手）**

1. 打开文件资源管理器，导航到 `C:\Program Files\nodejs\`
2. 看看这个文件夹里有没有 `node.exe` 文件
3. **如果有 `node.exe`** → 说明已安装，只是没添加到PATH，跳到下面的"手动添加环境变量"
4. **如果没有 `node.exe`** → 说明没安装，跳到"重新安装Node.js"

**方法B：手动添加环境变量（如果已安装但没加到PATH）**

1. 右键"此电脑" → **属性** → **高级系统设置** → **环境变量**
2. 在"系统变量"中找到 **Path**，双击打开
3. 点击 **新建**，添加Node.js的安装路径（通常是 `C:\Program Files\nodejs\`）
4. 点击 **确定** 保存
5. **关闭当前CMD窗口，重新打开**，再次输入 `node -v`

**方法C：重新安装Node.js（最稳妥）**

1. 访问 https://nodejs.org/ 下载 **Node.js 18.x LTS** 版本
2. 运行安装程序，一路点击 Next
3. **关键步骤**：在安装向导中，看到如下界面时：

```
Custom Setup
Select the way you want features to be installed.

☑ Node.js runtime
☑ npm package manager
☑ Add to PATH          ← ← ← 这一项必须勾选上！
☐ Online documentation shortcuts
```

4. 确保 **"Add to PATH"** 前面是 ☑ 勾选状态
5. 点击 Next → Install，等待安装完成
6. **关闭当前CMD窗口，重新打开**，再次输入 `node -v`

###### ❌ 情况二：`node -v` 能显示版本，但 `npm -v` 失败

**原因**：Node.js安装成功，但npm没有正确安装。

**解决方法：**

```bash
# 方法1：手动安装npm（在命令行中执行）

# 方法2：如果方法1不行，用Node.js自带的工具修复
# 先找到Node.js安装目录（通常是 C:\Program Files\nodejs\）
# 然后在该目录下打开命令行，执行：
cd "C:\Program Files\nodejs"
npm install -g npm

# 方法3：如果还不行，卸载Node.js重新安装
```

2. 右键"此电脑" → **属性** → **高级系统设置** → **环境变量**
3. 在"系统变量"中找到 **Path**，双击打开
4. 点击 **新建**，添加 `C:\Program Files\nodejs\`
5. 点击 **确定** 保存
6. **关闭当前CMD窗口，重新打开**，再次输入 `npm -v`

###### ❌ 情况四：PowerShell提示 `无法加载文件 npm.ps1，因为在此系统上禁止运行脚本`

**错误信息：**
```
npm : 无法加载文件 C:\Program Files\nodejs\npm.ps1，因为在此系统上禁止运行脚本。
```

**原因**：这是Windows PowerShell的安全策略（Execution Policy）问题，不是Node.js没安装好。PowerShell默认禁止运行 `.ps1` 脚本文件。

**解决方法一（推荐，最简单）：改用CMD（命令提示符）**

Node.js其实已经安装成功了，只是PowerShell的限制。换成CMD就好了：

1. 按 `Win + R` 键
2. 输入 `cmd`，回车
3. 在CMD窗口中输入：
```bash
node -v
npm -v
```

> 以后开发时建议都用CMD，或者在VS Code中把终端切换为CMD（点击终端右上角的下拉箭头 → 选择"命令提示符"）

**解决方法二（一劳永逸）：修改PowerShell执行策略**

如果你想继续使用PowerShell：

1. 按 `Win` 键，搜索 `PowerShell`
2. 右键 → **"以管理员身份运行"**
3. 在管理员PowerShell中输入：
```bash
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```
4. 输入 `Y` 确认
5. 关闭PowerShell，重新打开，再次输入 `npm -v` 验证

## 阶段二：基准引擎深度分析（第3-4周）

### 第3周：理解DependencyCheck核心架构

#### 3.1 核心引擎入口：Engine.java

`Engine.java` 是整个DependencyCheck的**大脑**，位于 `core/src/main/java/org/owasp/dependencycheck/Engine.java`。

它的核心工作流程：

```
1. 初始化阶段
   ├── 加载配置（Settings）
   ├── 发现所有分析器（AnalyzerService）
   └── 打开漏洞数据库连接（CveDB）

2. 扫描阶段（scan()）
   ├── 递归扫描指定目录
   ├── 根据文件类型匹配分析器
   └── 构建依赖树（Dependency）

3. 分析阶段（analyze()）
   ├── 按阶段执行所有分析器
   │   ├── INITIAL → 初始处理
   │   ├── INFORMATION_COLLECTION → 收集依赖信息
   │   ├── IDENTIFIER_ANALYSIS → 识别依赖标识
   │   ├── FINDING_ANALYSIS → 匹配漏洞
   │   └── FINAL → 最终处理
   └── 每个分析器处理特定类型的文件

4. 报告生成阶段（generateReport()）
   └── ReportGenerator生成HTML/XML/JSON报告
```

> **给实习生的理解**：可以把Engine想象成一个工厂流水线：
> 1. 先把项目文件"拆开"（扫描）
> 2. 然后逐个"检查"每个零件（分析）
> 3. 最后"打印"检查报告（报告生成）

#### 3.2 分析器体系（60+分析器）

分析器是DependencyCheck最核心的部分，每个分析器负责解析一种类型的文件。

**关键分析器分类：**

| 分析器类别 | 代表分析器 | 作用 |
|-----------|-----------|------|
| Java相关 | `JarAnalyzer` | 解析JAR包中的pom.xml、MANIFEST.MF |
| Maven相关 | `PinnedMavenInstallAnalyzer` | 解析Maven安装的依赖 |
| NPM相关 | `NodePackageAnalyzer` | 解析package.json |
| Python相关 | `PipAnalyzer`、`PipfileAnalyzer` | 解析Python依赖 |
| 漏洞匹配 | `CPEAnalyzer` | 将依赖与CPE（通用平台枚举）匹配 |
| 漏洞匹配 | `NvdCveAnalyzer` | 从NVD数据库匹配CVE漏洞 |
| 许可证分析 | 内置于多个分析器中 | 自动识别许可证类型 |
| 报告生成 | `FalsePositiveAnalyzer` | 过滤误报 |

**分析器执行顺序（AnalysisPhase）：**

```java
public enum AnalysisPhase {
    INITIAL,                    // 初始阶段
    INFORMATION_COLLECTION,     // 信息收集阶段1
    INFORMATION_COLLECTION2,    // 信息收集阶段2
    IDENTIFIER_ANALYSIS,        // 标识符分析
    POST_IDENTIFIER_ANALYSIS,   // 标识符后分析
    FINDING_ANALYSIS,           // 漏洞发现分析
    FINDING_ANALYSIS_PHASE2,    // 漏洞发现分析阶段2
    POST_FINDING_ANALYSIS,      // 漏洞发现后分析
    FINAL                       // 最终阶段
}
```

#### 3.3 漏洞数据源

| 数据源 | 全称 | 说明 |
|-------|------|------|
| NVD | National Vulnerability Database | 美国国家漏洞数据库，最权威的漏洞数据源 |
| OSV | Open Source Vulnerabilities | 开源漏洞数据库，覆盖更广 |
| RetireJS | Retire.js | 专门针对JavaScript库的漏洞库 |
| Known Exploited Vulnerabilities | CISA KEV | 美国CISA发布的已知被利用漏洞目录 |

#### 3.4 报告生成

`ReportGenerator.java` 支持生成多种格式的报告。**默认格式为 HTML**，如果不指定 `--format` 参数，会自动生成 HTML 报告。

| 格式 | 命令参数 | 类/方法 | 说明 |
|------|---------|---------|------|
| **HTML** ✅ **默认** | `--format HTML` 或不指定 | `generateReport()` | 最常用的可视化报告，浏览器可直接打开 |
| XML | `--format XML` | `generateXmlReport()` | 机器可读的XML格式 |
| JSON | `--format JSON` | `generateJsonReport()` | 机器可读的JSON格式 |
| SARIF | `--format SARIF` | `generateSarifReport()` | 静态分析结果交换格式（可用于CI/CD） |
| CSV | `--format CSV` | `generateCsvReport()` | 表格数据格式，可用Excel打开 |
| JUNIT | `--format JUNIT` | `generateJunitXmlReport()` | 单元测试报告格式（可用于Jenkins等） |
| ALL | `--format ALL` | 同时生成所有格式 | 一次性输出全部格式的报告 |

> **注意**：DependencyCheck **不支持原生 PDF 输出**。如果项目需要 PDF 格式的报告，需要通过 HTML 转 PDF 的方式实现（例如使用 wkhtmltopdf 或 Puppeteer）。

**同时生成多种格式：**
```bash
dependency-check --scan /path/to/project \
  --format HTML \
  --format JSON \
  --format CSV \
  --out ./reports
```

### 第6周：核心业务逻辑开发

#### 6.1 封装扫描引擎服务（ScanEngineService）

这是**最核心的类**，负责封装DependencyCheck的Engine。

```java
@Service
public class ScanEngineService {

    /**
     * 执行扫描任务
     *
     * @param scanTask 扫描任务信息
     * @param progressCallback 进度回调，用于实时更新进度
     */
    public ScanResult executeScan(ScanTask scanTask, Consumer<Integer> progressCallback) {
        // 1. 初始化Settings（从Nacos加载配置）
        Settings settings = new Settings();
        settings.setString(Settings.KEYS.DATA_DIRECTORY, dataDirectory);
        settings.setString(Settings.KEYS.NVD_API_KEY, nvdApiKey);

        // 2. 创建Engine实例
        try (Engine engine = new Engine(Engine.Mode.STANDALONE, settings)) {
            // 3. 执行扫描
            //    - 扫描指定目录
            //    - 排除指定路径
            engine.scan(projectDirectory);

            // 4. 执行分析
            //    - 按阶段执行所有分析器
            //    - 匹配漏洞和许可证
            engine.analyze();

            // 5. 生成报告
            //    - 根据指定格式生成报告
            engine.generateReport(reportPath, ReportGenerator.Format.HTML);

            // 6. 解析结果
            //    - 从Engine中获取所有依赖和漏洞信息
            List<Dependency> dependencies = engine.getDependencies();
            //    - 转换为自定义的ScanResult对象
            return convertToScanResult(dependencies);
        }
    }
}
```

> **关键理解**：`Engine` 实现了 `AutoCloseable` 接口，所以可以用 `try-with-resources` 自动释放资源。

#### 6.2 实现异步扫描

```java
@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean("scanExecutor")
    public Executor scanExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(2);       // 核心线程数
        executor.setMaxPoolSize(5);        // 最大线程数（支持5个并发扫描）
        executor.setQueueCapacity(10);     // 队列容量
        executor.setThreadNamePrefix("scan-");
        executor.initialize();
        return executor;
    }
}

@Service
public class ScanTaskService {

    @Async("scanExecutor")
    public CompletableFuture<Long> submitScan(Long projectId, ScanConfig config) {
        // 1. 创建扫描任务记录（状态：PENDING）
        ScanTask task = createTask(projectId, config);

        // 2. 更新状态为 RUNNING
        updateTaskStatus(task.getId(), "RUNNING");

        try {
            // 3. 执行扫描（调用ScanEngineService）
            ScanResult result = scanEngineService.executeScan(task, progress -> {
                // 更新进度
                updateTaskProgress(task.getId(), progress);
            });

            // 4. 保存扫描结果到数据库
            saveVulnerabilities(task.getId(), result.getVulnerabilities());
            saveLicenseIssues(task.getId(), result.getLicenseIssues());

            // 5. 生成报告并上传到OSS
            String reportUrl = reportService.generateAndUpload(task, result);

            // 6. 更新任务状态为 COMPLETED
            updateTaskStatus(task.getId(), "COMPLETED");

            return CompletableFuture.completedFuture(task.getId());
        } catch (Exception e) {
            // 7. 异常处理，更新状态为 FAILED
            updateTaskStatus(task.getId(), "FAILED");
            log.error("Scan failed for task {}", task.getId(), e);
            return CompletableFuture.failedFuture(e);
        }
    }
}
```

> **为什么需要异步？** 扫描一个项目可能需要几分钟甚至半小时，如果同步执行，用户发起的HTTP请求会一直等待，导致浏览器超时。异步执行可以让HTTP请求立即返回，用户通过轮询查看进度。

```java
@Service
public class ProjectService {

    public Long uploadProject(MultipartFile file, Long userId) {
        // 1. 验证文件格式（必须是ZIP）
        if (!file.getOriginalFilename().endsWith(".zip")) {
            throw new IllegalArgumentException("仅支持ZIP格式");
        }

        // 2. 保存ZIP文件到临时目录
        String tempDir = "./uploads/" + UUID.randomUUID();
        File zipFile = new File(tempDir + "/project.zip");
        file.transferTo(zipFile);

        // 3. 解压ZIP文件
        unzip(zipFile, tempDir + "/extracted");

        // 4. 创建项目记录
        Project project = new Project();
        project.setName(file.getOriginalFilename());
        project.setFilePath(tempDir + "/extracted");
        project.setUserId(userId);
        projectMapper.insert(project);

        return project.getId();
    }

    private void unzip(File zipFile, String destDir) {
        try (ZipInputStream zis = new ZipInputStream(new FileInputStream(zipFile))) {
            ZipEntry entry;
            while ((entry = zis.getNextEntry()) != null) {
                File newFile = new File(destDir, entry.getName());
                if (entry.isDirectory()) {
                    newFile.mkdirs();
                } else {
                    new File(newFile.getParent()).mkdirs();
                    Files.copy(zis, newFile.toPath());
                }
            }
        } catch (IOException e) {
            throw new RuntimeException("解压失败", e);
        }
    }
}
```

#### 8.2 扫描结果解析（从Engine结果转换为自定义对象）

```java
private ScanResult convertToScanResult(List<Dependency> dependencies) {
    ScanResult result = new ScanResult();
    List<Vulnerability> vulns = new ArrayList<>();
    List<LicenseIssue> licenses = new ArrayList<>();

    for (Dependency dep : dependencies) {
        // 提取漏洞信息
        for (VulnerableSoftware vs : dep.getVulnerableSoftware()) {
            for (Vulnerability v : vs.getVulnerabilities()) {
                Vulnerability vuln = new Vulnerability();
                vuln.setCveId(v.getName());
                vuln.setSeverity(v.getCvssV3() != null
                    ? v.getCvssV3().getBaseSeverity().name()
                    : "UNKNOWN");
                vuln.setCvssScore(v.getCvssV3() != null
                    ? v.getCvssV3().getBaseScore() : null);
                vuln.setDescription(v.getDescription());
                vuln.setDependencyName(dep.getDisplayFileName());
                vuln.setDependencyVersion(dep.getVersion());
                vulns.add(vuln);
            }
        }

        // 提取许可证信息
        for (License license : dep.getLicenses()) {
            LicenseIssue issue = new LicenseIssue();
            issue.setDependencyName(dep.getDisplayFileName());
            issue.setDependencyVersion(dep.getVersion());
            issue.setLicenseName(license.getName());
            issue.setLicenseUrl(license.getUrl());
            licenses.add(issue);
        }
    }

    result.setVulnerabilities(vulns);
    result.setLicenseIssues(licenses);
    result.setTotalDependencies(dependencies.size());
    return result;
}
```

### 11.3 常见问题排查

| 问题 | 可能原因 | 解决方法 |
|------|---------|---------|
| 扫描结果为空 | 项目路径不对 | 检查上传的文件是否解压正确 |

```dockerfile
# 使用多阶段构建

# 数据目录（挂载卷）

ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 12.2 创建docker-compose.yml

```yaml
version: '3.8'

services:

    environment:
      - SPRING_PROFILES_ACTIVE=prod

      - "80:80"
```

### 12.4 配置Nginx反向代理

```nginx
# nginx.conf
server {
    listen 80;
    server_name localhost;

    # 前端静态文件
    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端API代理
    location /api/ {
        proxy_pass http://dependency-analysis:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 大文件上传支持
        client_max_body_size 100m;
        proxy_read_timeout 300s;
    }

    # H2数据库控制台（仅开发环境）
    location /h2-console/ {
        proxy_pass http://dependency-analysis:8080;
        proxy_set_header Host $host;
    }
}
```

## 关键风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| NVD API 限流导致更新慢 | 扫描无法进行 | 高 | 申请API Key；配置代理；使用离线数据库 |
| JDK版本不兼容 | 编译失败 | 中 | 使用JDK 17 LTS；跳过不兼容的插件 |

| 扫描大型项目超时 | 用户体验差 | 中 | 异步执行；进度反馈；超时重试机制 |
| 前端与后端联调问题 | 功能不可用 | 中 | 先定义好API接口文档；使用Mock数据 |
