## 1. 项目概述

### 1.1 项目定位

基于 OWASP Dependency-Check 9.0.0 构建的集团内部轻量级**依赖安全与合规分析平台**。提供可视化的项目上传、漏洞扫描、许可证合规分析和多格式报告生成能力。

### 1.2 核心功能

| 功能模块 | 说明 |
|----------|------|
| **项目管理** | 上传 ZIP 格式项目压缩包，管理待扫描项目 |
| **扫描任务管理** | 异步扫描任务提交、状态跟踪、进度展示 |
| **漏洞检测** | 基于 NVD/OSV 数据源的已知漏洞（CVE）检测 |
| **许可证分析** | 自动识别依赖组件的开源许可证类型 |
| **多格式报告** | 支持 HTML、Excel、PDF 三种格式的报告生成与下载 |

### 1.3 基准引擎

所有核心分析能力基于 [OWASP Dependency-Check](https://github.com/jeremylong/DependencyCheck) 9.0.0，通过 `dependency-check-core` 库直接集成，使用 SPI（Service Provider Interface）自动发现分析器。

```
     ▼
Model 层       ──  Entity / DTO / VO
                             数据模型
```

### 3.3 异步任务执行模型

```
ScanTaskService.createTask()
    │
    ├── 1. 创建任务记录（状态: PENDING）
    │
    └── 2. 委托 scanTaskExecutorService.executeScan()  ← @Async("scanTaskExecutor")
            │                                                  运行在独立线程池
            ├── 更新状态: RUNNING
            ├── 调用 ScanEngineService.scan()
            │     ├── 创建 Engine(Settings)
            │     ├── engine.scan(scanPath)
            │     ├── engine.analyzeDependencies()
            │     ├── engine.getDependencies()    → Dependency[]
            │     └── engine.writeReports(...)    → HTML 报告
            ├── 保存 ScanResult 到数据库
            ├── 更新状态: COMPLETED (或 FAILED)
            └── engine.close()
```

> **关键设计**: `ScanTaskExecutorService` 独立于 `ScanTaskService`，因为 Spring AOP 的 `@Async` 注解必须通过代理对象调用才能生效。如果 `@Async` 方法在同一个类中被内部调用，代理无法拦截，会导致同步执行。

## 5. 数据库设计

使用 H2 嵌入式数据库，数据文件持久化至 `./data/dependency-check`，DDL 定义在 [schema.sql](dependency-check-web/src/main/resources/schema.sql)。

### 5.1 表结构

#### project（项目表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键（自增） |
| name | VARCHAR(255) | 项目名称 |
| description | VARCHAR(1000) | 项目描述 |
| file_path | VARCHAR(500) | 上传文件存储路径 |
| file_type | VARCHAR(50) | 文件类型（如 ZIP） |
| status | VARCHAR(20) | 状态（UPLOADED/SCANNING） |
| deleted | INT | 逻辑删除标记（0=正常, 1=已删除，MyBatis-Plus @TableLogic） |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

#### scan_task（扫描任务表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键（自增） |
| project_id | BIGINT | 关联项目 ID（FK） |
| status | VARCHAR(20) | 状态：PENDING → RUNNING → COMPLETED/FAILED |
| progress | INT | 进度百分比（0-100） |
| total_dependencies | INT | 扫描到的依赖总数 |
| vulnerable_dependencies | INT | 存在漏洞的依赖数 |
| error_message | VARCHAR(2000) | 失败时的错误信息 |
| started_at | DATETIME | 扫描开始时间 |
| completed_at | DATETIME | 扫描完成时间 |
| created_at | DATETIME | 创建时间 |

**状态流转**: `PENDING` → `RUNNING` → `COMPLETED` / `FAILED`

#### scan_result（扫描结果表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键（自增） |
| task_id | BIGINT | 关联任务 ID（FK） |
| dependency_name | VARCHAR(500) | 依赖名称 |
| dependency_version | VARCHAR(100) | 依赖版本 |
| file_path | VARCHAR(1000) | 依赖文件路径 |
| cve_id | VARCHAR(50) | CVE 编号（如 CVE-2021-44228） |
| cvss_score | DECIMAL(3,1) | CVSS 评分（优先 v3，其次 v2） |
| severity | VARCHAR(20) | 严重等级（CRITICAL/HIGH/MEDIUM/LOW） |
| description | VARCHAR(2000) | 漏洞描述 |
| license_name | VARCHAR(200) | 许可证名称 |
| is_vulnerable | BOOLEAN | 是否存在漏洞 |

### 5.2 索引

- `idx_scan_task_project_id` ON `scan_task(project_id)`
- `idx_scan_result_task_id` ON `scan_result(task_id)`
- `idx_scan_result_cve_id` ON `scan_result(cve_id)`
- `idx_scan_result_severity` ON `scan_result(severity)`

## 7. 核心业务流程

### 7.1 完整扫描流程

```
用户上传 ZIP 项目
    │
    ▼
ProjectService.uploadProject(file, name, description)
    ├── 保存文件到 ./uploads/
    ├── 解压到 ./uploads/{projectId}/
    ├── 创建 Project 记录（status=UPLOADED）
    └── 返回 ProjectDTO
    │
    ▼
用户点击「开始扫描」
    │
    ▼
ScanTaskController.createTask(projectId)
    │
    ▼
ScanTaskService.createTask(projectId)
    ├── 校验项目存在
    ├── 创建 ScanTask 记录（status=PENDING, progress=0）
    ├── 委托 scanTaskExecutorService.executeScan(taskId, filePath, reportDir)
    └── 立即返回 ScanTaskDTO
    │
    ▼ (异步线程池执行)
ScanTaskExecutorService.executeScan()
    ├── 更新 status=RUNNING, started_at=now
    ├── 调用 scanEngineService.scan(scanPath, reportDir)
    │     ├── 创建 Engine(Settings)
    │     │     ├── 设置 DATA_DIRECTORY=./data/dc-cache (漏洞数据库缓存)
    │     │     └── 设置 NVD_API_KEY（如果已配置）
    │     ├── engine.scan(scanPath)           ← 文件分析，匹配分析器
    │     ├── engine.analyzeDependencies()    ← 漏洞/许可证分析
    │     ├── engine.getDependencies()         ← 获取 Dependency[]
    │     ├── engine.writeReports(...)         ← 生成 HTML 报告
    │     └── engine.close()                   ← 释放数据库连接
    ├── convertToScanResults() → 每个漏洞一条 ScanResult 记录
    ├── 批量插入 ScanResult
    └── 更新 status=COMPLETED, progress=100, 统计信息
    │
    ▼
用户轮询 GET /api/tasks/{id} 获取进度
    │
    ▼
用户访问 GET /api/reports/{taskId}?format=html|excel|pdf 查看报告
```

### 7.2 扫描引擎集成细节（[ScanEngineService.java](dependency-check-web/src/main/java/com/alibaba/dependencycheck/service/ScanEngineService.java)）

核心代码位于 `ScanEngineService.scan()` 方法：

```java
// 1. 创建 Settings 并配置
Settings settings = new Settings();
settings.setString(Settings.KEYS.DATA_DIRECTORY, dataDirectory);  // ./data/dc-cache
settings.setString(Settings.KEYS.NVD_API_KEY, nvdApiKey);         // 可选

// 2. 构建引擎 — SPI 自动发现所有分析器
Engine engine = new Engine(Engine.Mode.STANDALONE, settings);

// 3. 文件扫描 — 根据文件类型匹配分析器
engine.scan(scanPath);

// 4. 依赖分析 — 按阶段执行所有分析器，匹配漏洞数据
engine.analyzeDependencies();

// 5. 获取结果
Dependency[] deps = engine.getDependencies();

// 6. 生成报告
engine.writeReports("dependency-check", reportDirectory, "HTML");

// 7. 必须释放
engine.close();
```

### 7.3 结果转换逻辑

`ScanTaskExecutorService.convertToScanResults()` 将 OWASP Dependency 对象转为 `ScanResult` 实体：

- **无漏洞** → 生成 1 条记录（`is_vulnerable=false`），记录依赖名/版本/许可证
- **有漏洞** → 每个 CVE 生成 1 条记录（`is_vulnerable=true`），提取 CVE ID / CVSS评分（优先v3）/ 严重等级 / 描述 / 许可证

> **说明**: `max-scan-threads` 映射到 `Settings.KEYS.MAX_DOWNLOAD_THREAD_POOL_SIZE`，控制 NVD 数据下载的并发线程数。OWASP 内部分析线程由 `Runtime.availableProcessors()` 控制，不受此配置影响。

```yaml

  upload-dir: ./uploads        # 项目 ZIP 包上传目录
  report-dir: ./reports        # 扫描报告输出目录
```

### 8.6 异步任务线程池

```yaml
task:
  pool:
    core-pool-size: 2           # 核心线程数
    max-pool-size: 4            # 最大线程数
    queue-capacity: 10          # 等待队列容量
    thread-name-prefix: scan-task-
```

此配置控制 `@Async("scanTaskExecutor")` 线程池，负责 Web 层的异步任务调度，**不控制 OWASP 引擎内部的扫描线程**。

### 8.7 CORS 跨域配置

```yaml
app:
  cors:
    allowed-origin: http://localhost:5173   # CORS 允许的前端来源（7/3 C4 修复：已从硬编码改为配置读取）
```

> **注意**: 当前 Vite 开发服务器实际端口为 `3000`（见 [vite.config.js](dependency-check-web/frontend/vite.config.js)），CORS 默认值 `5173` 与 Vite 默认端口一致。开发环境通过 Vite 代理转发 API 请求，CORS 配置主要服务于生产部署场景。

### 9.3 首次使用完整操作流程

#### 步骤 1: 准备 NVD 漏洞数据库（重要！）

OWASP Dependency-Check 需要同步 NVD（National Vulnerability Database）数据才能检测漏洞。首次启动时会自动开始下载，数据量约 **20-30 GB**，耗时可能 **30 分钟到数小时**，取决于网络环境。

**⚠️ 由于国内网络限制，强烈建议配置 NVD API Key 加速下载：**

1. 前往 [NVD API Key Request](https://nvd.nist.gov/developers/request-an-api-key) 免费申请
2. 设置环境变量后启动：
```bash
# Windows (Git Bash / PowerShell)
export NVD_API_KEY="your-api-key-here"

# 或在启动时指定
mvn spring-boot:run -Dspring-boot.run.arguments="--dependency-check.nvd-api-key=your-api-key"
```

3. 漏洞数据库存放于 `./data/dc-cache/`，首次下载后后续启动无需重新下载（仅增量更新）

#### 步骤 2: 上传项目

1. 打开浏览器访问 `http://localhost:3000`（前端）或使用 curl：
```bash
curl -X POST http://localhost:8080/api/projects \
  -F "file=@/path/to/your-project.zip" \
  -F "name=测试项目" \
  -F "description=用于测试的项目"
```

#### 步骤 3: 启动扫描

```bash
# 替换 {projectId} 为步骤2返回的项目ID
curl -X POST "http://localhost:8080/api/tasks?projectId={projectId}"
```

返回任务信息后，扫描在后端异步执行。

#### 步骤 4: 检查扫描进度

```bash
# 替换 {taskId} 为步骤3返回的任务ID
curl http://localhost:8080/api/tasks/{taskId}
```

当 `status` 变为 `COMPLETED` 时扫描完成。

#### 步骤 5: 查看漏洞结果

```bash
curl http://localhost:8080/api/tasks/{taskId}/results
```

返回所有依赖的漏洞和许可证信息。

#### 步骤 6: 下载报告

```bash
# HTML 报告
curl "http://localhost:8080/api/reports/{taskId}?format=html" -o report.html

# Excel 报告
curl "http://localhost:8080/api/reports/{taskId}?format=excel" -o report.xlsx

# PDF 报告
curl "http://localhost:8080/api/reports/{taskId}?format=pdf" -o report.pdf
```

### 9.4 使用测试项目验证功能

项目提供了 3 个标准测试项目：

1. **Spring PetClinic** (`实习专用/spring-petclinic/`) — 基础功能验证
2. **Nacos Example** (`实习专用/spring-cloud-alibaba/...`) — 多模块扫描
3. **Damn Vulnerable Spring Boot App** (`damn-vulnerable-spring-boot-app/`) — 漏洞检测准确性

将一个项目目录打包为 ZIP：

```bash
# 以 PetClinic 为例
cd 实习专用/spring-petclinic
zip -r ../../spring-petclinic.zip .
```

然后通过 Web 界面上传 `spring-petclinic.zip` 即可测试。

### 9.5 JVM 参数调优

对于大型项目，建议调整 JVM 参数防止 OOM：

```bash
# Maven 启动
MAVEN_OPTS="-Xms512m -Xmx4g" mvn spring-boot:run

# JAR 启动
java -Xms512m -Xmx4g -jar target/dependency-check-web-1.0.0-SNAPSHOT.jar
```

### 9.6 常见启动问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 端口 8080 被占用 | 其他程序占用 | 修改 `application.yml` 中 `server.port`，或杀掉占用进程 |
| NVD 下载失败 | 网络不通 | 配置代理，或手动下载 NVD 数据放入 `./data/dc-cache/` |
| 前端 npm install 失败 | 网络/Node 版本 | 设置 npm 镜像 `npm config set registry https://registry.npmmirror.com` |
| Java 版本不匹配 | JDK 版本 < 17 | 安装 JDK 17+ 并配置 JAVA_HOME |
| OutOfMemoryError | 扫描大项目时堆空间不足 | 增加 `-Xmx` 参数（见 9.5） |

## 11. 已知问题与限制

### 11.1 ~~max-scan-threads 未生效~~ ✅ 已修复（6/29 B1-03）

- **位置**: [application.yml:39](dependency-check-web/src/main/resources/application.yml#L39) → [ScanEngineService.java:104](dependency-check-web/src/main/java/com/alibaba/dependencycheck/service/ScanEngineService.java#L104)

- **当前状态**: ✅ 配置已生效，控制 NVD 数据下载线程池大小

### 11.2 CORS 端口注意事项

- **描述**: `application.yml` 中 `app.cors.allowed-origin` 默认值为 `http://localhost:5173`（Vite 默认端口），但本项目 `vite.config.js` 配置的开发端口是 `3000`
- **影响**: 开发环境通过 Vite 代理转发 `/api` 请求，CORS 不参与；生产部署时需将 `allowed-origin` 设为实际前端域名
- **修复**: 7/3 C4 修复已将 origin 从硬编码改为 `@Value` 配置读取，可在 `application.yml` 中按环境切换

### 11.3 无 Docker 部署文件

- **描述**: PRD 要求提供 Docker Compose 一键部署，当前尚未实现
- **影响**: 部署需要手动配置环境
- **计划**: 阶段六（第12周）

### 11.4 无用户认证

- **描述**: PRD 要求简单的用户登录（固定账号密码），当前未实现
- **影响**: 系统无访问控制
- **优先级**: P2（非核心功能）

### 11.5 ~~扫描进度非实时~~ ✅ 已修复（6/30 B3-09）

- **原描述**: 扫描进度只有 0 和 100，无中间值
- **修复内容**: `ScanTaskExecutorService.executeScan()` 添加三个进度点 — 10%（开始扫描）、90%（保存结果中）、100%（完成）
- **当前状态**: ✅ 前端轮询可看到进度阶段性变化

### 11.6 漏洞数据库首次下载耗时长

- **描述**: NVD 数据首次全量同步约 20-30 GB，国内网络可能数小时
- **缓解**: 申请 NVD API Key 可显著提速；后续增量更新仅需数分钟
- **已配置**: `ScanEngineService` 设置了 CONNECTION_TIMEOUT=30s、CONNECTION_READ_TIMEOUT=600s、ANALYSIS_TIMEOUT=300s

### 11.7 阿里云 OSS 集成未实现

- **描述**: PRD 要求报告存储于阿里云 OSS，当前报告存储在本地文件系统 `./reports/`
- **影响**: 多实例部署时报告文件无法共享
- **计划**: D4 子阶段（第8周）
