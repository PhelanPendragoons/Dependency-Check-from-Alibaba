```
analyzeDependencies()
  ├── initializeAndUpdateDatabase() → 更新 NVD 数据库

  │   └── 遍历每个阶段的 Analyzer
```

##### 分析阶段（AnalysisPhase）共15个阶段
| 阶段 | 代表分析器 | 作用 |
|------|-----------|------|
| INITIAL | ArchiveAnalyzer | 初始处理（解压归档文件） |

| POST_INFORMATION_COLLECTION1~3 | DependencyMergingAnalyzer, HintAnalyzer, VersionFilterAnalyzer | 后处理 |
| PRE_IDENTIFIER_ANALYSIS | NpmCPEAnalyzer | 预标识符分析 |
| IDENTIFIER_ANALYSIS | CPEAnalyzer | CPE匹配 |
| POST_IDENTIFIER_ANALYSIS | CpeSuppressionAnalyzer, FalsePositiveAnalyzer | 后标识符分析 |

| FINAL | DependencyBundlingAnalyzer, UnusedSuppressionRuleAnalyzer | 最终处理 |

##### 关键设计模式
- **SPI（Service Provider Interface）**：通过 `ServiceLoader<Analyzer>` 动态加载所有分析器
- **策略模式**：每个分析器实现 `Analyzer` 接口，按阶段执行
- **多线程并行**：支持并行处理的分析器使用 `Executors.newFixedThreadPool(maxThreads)`
- **模板方法**：`AbstractFileTypeAnalyzer` 提供通用文件分析模板

```
analyzeDependency(dependency, engine)
  ├── collectClassNames() → 收集 JAR 中的类名
  ├── 检查是否为 macOS 元数据文件或无效 ZIP → 是则移除依赖
  ├── parseManifest() → 解析 MANIFEST.MF
  │   ├── 提取 Implementation-Title, Implementation-Version
  │   ├── 提取 Bundle-Name, Bundle-Version, Bundle-Description
  │   └── 过滤 IGNORE_KEYS 和 IGNORE_VALUES
  ├── analyzePOM() → 解析 pom.xml / pom.properties
```

- **去重机制**：通过 SHA1 哈希判断是否已存在相同依赖
- **排除机制**：排除 macOS 元数据文件（`._` 前缀）和无效 ZIP 文件

#### 2. 设计数据库表结构（schema.sql）✅
- **操作**：根据项目完成计划 A3 节的表结构设计，创建 3 张核心表
- **3 张核心表**：
  - `project` — 项目表（名称、描述、文件路径、文件类型、状态、创建时间等）
  - `scan_task` — 扫描任务表（关联项目、状态、进度、依赖总数、漏洞数、错误信息、开始/完成时间等）
  - `scan_result` — 扫描结果表（关联任务、依赖名称/版本、文件路径、CVE编号、CVSS评分、严重等级、漏洞描述、许可证信息等）
- **索引设计**：
  - `project` 表：`name` 字段唯一索引（防止重复项目名）
  - `scan_task` 表：`project_id` 索引（加速按项目查询任务）
  - `scan_result` 表：`task_id` 索引（加速按任务查询结果）
- **外键约束**：
  - `scan_task.project_id` → `project.id`（级联删除）
  - `scan_result.task_id` → `scan_task.id`（级联删除）
- **设计决策**：
  - 使用 `VARCHAR(20)` 存储状态枚举（PENDING/RUNNING/COMPLETED/FAILED），而非 INT 枚举，提高可读性
  - `cvss_score` 使用 `DECIMAL(3,1)` 精确存储（如 9.8），避免浮点数精度问题
  - `is_vulnerable` 使用 `BOOLEAN` 字段标记是否有漏洞，便于快速筛选
  - 所有表均包含 `created_at` 和 `updated_at` 时间戳字段
- **遇到的问题**：
  - ❌ **DECIMAL(3,1) 能否存 10.0？** 一开始不确定 3 位总长度够不够，查了 H2 文档确认 DECIMAL(3,1) 范围是 -99.9 到 99.9，10.0 只需要 2 位整数 + 1 位小数，没问题
  - ❌ **VARCHAR(4000) 够不够存漏洞描述？** 看了几个真实 CVE 的描述，大部分在 500 字以内，4000 应该够了

      ```java
      }
      ```

    - **原因**：Spring Boot 默认最大上传文件大小为 1MB

    - **解决**：查阅 v9.0.0 API 文档，使用 `vuln.getCvssV3().getCvssData().getBaseScore()` 获取评分

    - **原因**：`@Async` 注解需要 `@EnableAsync` 配置，且需要在不同的 Bean 中调用才能生效（Spring AOP 代理限制）
    - **解决**：确认 `AsyncConfig.java` 已配置 `@EnableAsync`，并将 `executeScan()` 方法放在单独的 Service 类中
- **关键设计**：
  - 使用 `@Async("scanTaskExecutor")` 确保扫描在独立线程中执行
  - 任务状态变更记录时间戳（startedAt、completedAt）
  - 异常时记录错误信息到 `errorMessage` 字段
