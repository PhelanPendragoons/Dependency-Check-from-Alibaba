### 2.2 核心模块源码分析

#### 2.2.1 Engine 核心引擎架构

分析了 `Engine.java`（1365行）的三大核心流程：

**① 初始化流程**
```
Engine 构造函数 → initializeEngine() → loadAnalyzers()
  → AnalyzerService（SPI 机制加载 Analyzer）
  → 按 AnalysisPhase 分组存入 EnumMap
```

**② 扫描流程**
```
scan(path) → scan(file)
  → 目录则递归扫描
  → 文件则创建 Dependency 对象
  → 计算 SHA1 哈希（去重）
```

**③ 分析流程**
```
analyzeDependencies()
  → initializeAndUpdateDatabase() 更新 NVD
  → 遍历 15 个 AnalysisPhase
  → 每个阶段并行执行 Analyzer
  → 多线程 AnalysisTask
```

#### 2.2.2 15 个分析阶段体系

| 阶段 | 代表分析器 | 作用 |
|------|-----------|------|
| INITIAL | ArchiveAnalyzer | 解压归档文件 |
| INFORMATION_COLLECTION | JarAnalyzer, CentralAnalyzer 等 30+ | 收集依赖信息 |
| IDENTIFIER_ANALYSIS | CPEAnalyzer | CPE 匹配 |
| FINDING_ANALYSIS | NvdCveAnalyzer, RetireJsAnalyzer | 漏洞发现 |
| FINDING_ANALYSIS_PHASE2 | OssIndexAnalyzer | Sonatype OSS Index 查询 |
| POST_FINDING_ANALYSIS | KnownExploitedVulnerabilityAnalyzer | KEV 标记 |
| FINAL | DependencyBundlingAnalyzer | 依赖打包 |

#### 2.2.3 8 个核心分析器深度分析

| 分析器 | 文件行数 | 核心功能 |
|--------|:--------:|----------|
| **ArchiveAnalyzer** | - | 解压 ZIP/JAR/WAR/TAR 等 20+ 种归档格式，递归提取内部依赖 |
| **JarAnalyzer** | 1450行 | 解析 MANIFEST.MF、pom.xml、包名，从三个维度收集证据 |
| **CentralAnalyzer** | - | 通过 SHA1 哈希查询 Maven Central，获取精确 GAV 坐标 |
| **CPEAnalyzer** | 1416行 | 使用 Lucene 索引进行 CPE 模糊匹配，按置信度逐级搜索 |
| **NvdCveAnalyzer** | 203行 | 将 CPE 标识符与 NVD 数据库中的 CVE 记录匹配 |
| **OssIndexAnalyzer** | 403行 | 批量查询 Sonatype OSS Index API，使用 PURL 标识依赖 |
| **FalsePositiveAnalyzer** | - | 内置误报规则，移除虚假 CPE 匹配，添加漏报 CPE |
| **ReportGenerator** | 594行 | 使用 Apache Velocity 模板引擎，支持 9 种报告格式 |

#### 2.2.4 关键设计模式发现

- **SPI 机制**：通过 `ServiceLoader<Analyzer>` 动态加载所有分析器
- **策略模式**：每个分析器实现 `Analyzer` 接口，按阶段执行
- **多线程并行**：使用 `Executors.newFixedThreadPool(maxThreads)` 并行分析
- **模板方法**：`AbstractFileTypeAnalyzer` 提供通用文件分析模板
- **证据模型**：从 Manifest、POM、包名三个维度收集证据，按置信度分级

## 三、遇到的问题与解决方案

### 3.1 环境配置问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| SpotBugs 编译失败 | JDK 26 不支持 SpotBugs 4.8.6.6 | 添加 `-Dspotbugs.skip=true` 跳过 |
| PowerShell 参数解析错误 | PowerShell 对 `-D` 参数处理不同 | 用双引号包裹参数 `"-Dspotbugs.skip=true"` |
| Checkstyle 找不到配置文件 | 中文路径导致路径解析失败 | 添加 `-Dcheckstyle.skip=true` 跳过 |
| Maven 中央仓库下载慢 | 国内网络环境限制 | 配置阿里云镜像（settings.xml） |
| Gradle 6.x 不支持 JDK 17 | Gradle 版本过旧 | 升级到 Gradle 7.6.4 |
| Spring Boot 1.x 插件不兼容 | 使用了已废弃的 `runtime` 配置 | 升级到 Spring Boot 2.7.18 |

### 3.2 JDK 26 兼容性问题 — 已解决 ✅

**问题**：JDK 26 与多个工具存在 SSL 兼容性问题（`SSLHandshakeException: handshake_failure`），导致 NVD API 和 Sonatype OSS Index 均无法连接。

**解决方案**：系统已安装 JDK 17（Zulu 17.0.14 LTS），通过修改用户 PATH 环境变量将 JDK 17 的 bin 目录置于最前面，使系统默认使用 JDK 17。

**验证结果**：
- ✅ JDK 17 切换成功：`openjdk version "17.0.14" 2025-01-21 LTS`
- ✅ DependencyCheck 扫描成功（96秒），报告正常生成
- ⚠️ **Sonatype OSS Index 仍不可用** — 原因已查明：Sonatype OSS Index API 现在**需要 API Key（用户名+密码）认证**，返回 `401 Unauthorized`，不再是匿名可用的服务
- ⚠️ NVD API 仍需要 API Key 才能高效更新

**永久生效说明**：PATH 修改已写入用户环境变量，新打开的 CMD/PowerShell 窗口会自动使用 JDK 17。当前 CMD 会话需手动设置：
```cmd
set JAVA_HOME=C:\Program Files\Java\zulu17.56.15-ca-jdk17.0.14-win_x64
set PATH=C:\Program Files\Java\zulu17.56.15-ca-jdk17.0.14-win_x64\bin;%PATH%
```

### 3.3 NVD API Key 与 Sonatype OSS Index 问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| NVD 数据库下载慢 | 未配置 NVD API Key，354,495 条记录下载极慢 | 申请 NVD API Key（免费） |
| Sonatype OSS Index 不可用 | API 需要用户名+密码认证（`--ossIndexUsername` + `--ossIndexPassword`） | 注册 Sonatype OSS Index 账号获取 API Key |
| Hosted Suppressions 文件下载失败 | 网络连接问题 | 不影响扫描，仅影响误报过滤 |

**建议**：
1. 申请 NVD API Key：https://nvd.nist.gov/developers/request-an-api-key
2. 注册 Sonatype OSS Index：https://ossindex.sonatype.org/ 获取 API 凭证
3. 使用以下命令配置认证：
```bash
dependency-check.bat --noupdate --scan "目标.jar" --out "报告目录" --format HTML ^
  --ossIndexUsername "your-username" --ossIndexPassword "your-password"
```
