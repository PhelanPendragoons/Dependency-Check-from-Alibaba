### 幻灯片 3 → 4 — 项目背景与简介（50 秒）

> 为什么做这个项目。
>
> 现代 Java 应用 80% 以上的代码来自第三方开源组件——Log4Shell 这种满分漏洞一旦爆发，攻击者通过一条日志就能拿到服务器权限。而人工排查依赖漏洞极其耗时——依赖树层级深、传递性依赖多，逐条比对 CVE 数据库几乎不可能覆盖全。
>
> 业界是有标准工具的——OWASP Dependency-Check。但它是命令行工具：环境配置复杂、NVD 数据下载有门槛、输出是纯文本日志，普通开发者很难上手。
>
> 我的方案是把 ODC 引擎以 library 方式嵌入 Spring Boot，做成一个 Web 平台——上传 ZIP 包、一键扫描、可视化结果、支持三种格式报告下载。把安全检查从「专家工具」变成「任何人 5 分钟能用的平台」。

**可能的追问**：
- Q: "为什么不从零开发扫描引擎？"
- A: ODC 是业界标准，NVD 数据映射的准确性经过 10 年验证。重写引擎不是实习的目标——我的价值是把成熟的引擎能力「产品化」，补齐可视化、任务管理、报告生成这些引擎不具备的生产能力。

> 下面我从 PPT 切到系统，直接操作给大家看。

> 这页讲系统长什么样。从左到右一条完整的流水线——
>
> 浏览器端 Vue 3 上传项目 ZIP 包，到达后端 Spring Boot REST API。此时立即返回——扫描被丢进异步线程池，前端每 3 秒轮询进度，用户不需要等。
>
> 引擎层封装 ODC 9.0.0。漏洞数据库用的是 250MB 的 NVD 本地缓存——接下来现场演示大家会看到，全程不联网也能完成扫描。
>
> 扫描结果落 H2 数据库持久化。右下角——报告可一键导出 HTML、Excel、PDF 三种格式。最右侧是阿里云 OSS 云存储的弹性适配——默认本地存储，配环境变量就切云端，对业务代码透明。
>
> 下面是六大核心特性——
>
> 第一，安全的 ZIP 上传。解压时做了路径规范化和前缀白名单校验，防止恶意文件名里的 `../` 逃逸上传目录——这是著名的 Zip Slip 漏洞，Snyk 2018 年公布的，我做了三层防护。
>
> 第二，异步任务调度。上传完立即返回，扫描在线程池里后台跑，前端每 3 秒轮询进度条——用户不用盯着等。
>
> 第三，漏洞按严重等级分级统计。仪表盘上 CRITICAL / HIGH / MEDIUM / LOW 四个卡片独立显示，一眼看出哪些项目需要优先修。
>
> 第四，三格式报告一键下载。HTML 在线看、Excel 做数据分析、PDF 归档留存——同一个接口，传不同 format 参数即可，业务代码不关心生成细节。
>
> 第五，Swagger 在线接口文档8080端口。所有 REST API 自动生成可交互文档，直接在浏览器里调接口测试，不用切 Postman。对接 CI/CD 流水线时这就是 API 契约。
>
> 第六，存储弹性降级。默认本地文件存储，配三个 OSS 环境变量就切到阿里云对象存储。OSS 不可用时自动退回本地——对上层代码完全透明。

**可能的追问**：
- Q: "为什么用 H2 而不是 MySQL？"
- A: H2 是零依赖的文件数据库，开箱即用。MyBatis-Plus 屏蔽了 SQL 方言差异，迁移到 MySQL 只需改一个连接串，代码零改动。选择 H2 是为了降低运行门槛，不是技术限制。
- Q: "250MB NVD 缓存怎么维护？"
- A: 目前是预热好的离线演示方案。生产环境支持三种模式——API 直连（标准路径）、内部镜像服务（NVD 数据一次同步全公司复用）、离线缓存（当前演示模式）。通过环境变量在部署时透明切换。

### 幻灯片 10 — 技术难点（留在 PPT 上，演示完回到这页讲）（60 秒）

> 三个最有含金量的技术攻坚——
>
> 第一，NVD 数据源 Cloudflare 拦截。现象是 curl 直连 200，但 Java 客户端恒 520。根因是 Cloudflare 按 TLS 指纹识别非浏览器客户端并阻断。解决方案是切换 DC 引擎的 datafeed 通道，走镜像源 + 时间戳规范化 + 本地伺服，最终实现全离线扫描——接下来演示的扫描完全不需要联网。
>
> 第二，@Async 异步注解失效。同类方法内部调用绕过了 Spring AOP 代理，异步线程池不生效，任务变成同步执行。解法是把执行逻辑拆到独立 Service，强制走代理层。
>
> 第三，Zip Slip 路径穿越防护。恶意 ZIP 文件名嵌入 `../` 来逃逸上传目录。我做了路径规范化、前缀白名单、条目类型校验三层防护。

**可能的追问**：
- Q: "Cloudflare 拦截怎么排查出来的？"
- A: curl 能通 → Java 不通 → 排除 Key 和网络问题 → 对比两端的请求特征 → 发现 Java 的 TLS 指纹被识别 → 换协议通道绕过。关键的排查思想是「逐层排除变量」。
- Q: "为什么不直接换 VPN 节点解决？"
- A: 换了多个节点都是 520——问题不在 IP 而在客户端指纹。升级 open-vulnerability-clients 到最新版也无效，因为拦截发生在 TLS 握手层，不是 HTTP 层。

## 四、高频 QA 速查

| 可能的问题 | 回答要点 |
|-----------|---------|
| "为什么没做用户认证？" | 聚焦核心能力——依赖扫描和漏洞检测。认证鉴权架构上预留了扩展点（Spring Security 一句话引入），但实习优先级在扫描能力而非基础设施。对标同类开源工具，无认证不影响核心功能验证 |
| "H2 数据库重启会丢数据吗？" | 不会。用的是文件持久化模式（`jdbc:h2:file:`），重启后数据完整保留。优点是零依赖开箱即用。MyBatis-Plus 已屏蔽 SQL 差异，迁移到 MySQL 只改连接配置 |
| "NVD 数据更新频率？" | NVD 每 2 小时发布一次增量更新。DC 引擎默认每 4 小时检查一次。当前演示模式关闭了自动更新，生产环境打开后自动同步 |
| "能扫什么语言的项目？" | ODC 引擎原生支持 Java、.NET、Python、Node.js、Ruby 等。封装成 Web 平台后，语言支持取决于引擎版本，业务代码不受影响 |
| "扫描一个项目要多久？" | 首次扫描 1-2 分钟（含依赖分析 + CPE 匹配）。二次扫描只需 10-30 秒，因为 NVD 数据已在本地缓存中 |
| "61MB 的 ZIP 上传不慢吗？" | 上传 5-8 秒——瓶颈在网络带宽而非系统。解压和扫描是异步的，用户上传完就能离开页面，完成后通知 |
| "有考虑 CI/CD 集成吗？" | 暴露了完整的 REST API + Swagger 文档。CI 流水线只需 curl 调用 `/api/projects` 上传 + `/api/tasks` 触发扫描 + `/api/reports` 下载报告，就能嵌入到 Jenkins/GitLab CI 中 |
| "最大的技术挑战是什么？" | NVD 数据源被 Cloudflare 拦截。curl 能通，Java 不通——花了半天定位到客户端 TLS 指纹差异，最终切换 datafeed 协议通道解决。体现了「分步排除变量」的排查方法论 |
| "你觉得这个项目还有什么不足？" | 一是认证鉴权还没做，二是 H2 不适合高并发多用户场景，三是 DC 引擎版本可从 9.0.0 升级到 12.x。这些已经列入后续迭代计划 |
| "最有收获的是什么？" | 不是某个技术点，而是独立负责一个完整项目全流程的经验——从需求分析、技术选型、开发测试、安全审计、文档交付到版本发布——这个闭环让我真正理解了什么叫做「工程化思维」 |

## 🚨 现场应急

| 问题 | 措施 |
|------|------|
| 后端启动失败 | 检查 Java 17+：`java -version`；删 `data/dependency-check.mv.db` 重试 |
| 前端白屏 | 检查 Node 18+；重跑 `npm run dev` |
| 端口占用 | `netstat -ano \| findstr 8080` → `taskkill /PID xxx /F` |
| NVD 缓存损坏 | 删 `data/dc-cache/`，重新跑预热流程（有备份 zip） |
| 万用兜底 | 提前截仪表盘和报告截图；如果现场环境崩了，直接看图讲，保证不冷场 |
