# 深度问答速查 — 导师可能追问的技术细节

> 配合 [演示操作稿.md](演示操作稿.md) 第四章使用。以下每题的回答都基于项目真实代码/日志。

---

## 一、架构设计类

### Q: "你的异步任务怎么保证不丢？"
**A**: 任务创建时同步落库（`scan_task` 表），状态为 PENDING。异步线程池取任务、执行、更新状态——整个过程状态机流转都有数据库兜底。即使 JVM 崩溃重启，未完成的任务停留在 PENDING / RUNNING 状态，不会消失。**但不支持自动重试**（这是已知限制，文档里有写）。

### Q: "线程池参数怎么选的？"
**A**: 默认 4 个核心线程，对应 NVD 下载线程池大小。选择基于两个约束：一是 NVD API 有速率限制（有 Key 时 50 req/30s），线程太多会被限流；二是 ODC 引擎的 `analyzeDependencies()` 内部已使用 `Runtime.availableProcessors()` 并行分析，外层不需要再放大。

### Q: "H2 文件数据库多任务并发安全吗？"
**A**: H2 文件模式有写锁——同一时刻只有一条连接写。但你们的扫描引擎内部 NVD 缓存也是 H2 文件，所以**两个 H2 实例不共享锁**。实测两个任务同时扫时 NVD 读操作是线程安全的，写操作（落扫描结果）走自己的 H2，不存在竞争。

## 二、安全类

### Q: "ZIP 上传还有别的攻击面吗？"
**A**: 三种：Zip Bomb（压缩比极高的恶意文件——目前没做压缩率检测，是 ISSUES.md 里的待办）；ZIP Slip（已做三层防护）；文件类型伪装（上传 .exe 改名为 .zip → 当前靠 ZIP 格式校验拒绝，但没做深度文件魔术字检测）。这些在生产就绪清单里。

### Q: "NVD API Key 你怎么管理的？"
**A**: 通过环境变量 `${NVD_API_KEY:}` 注入，默认值为空。本地测试用 `nvd-api.properties` 存 Key，已在 `.gitignore` 排除，Git 历史中无泄露。`application.yml` 提交前的默认值是空的。

### Q: "扫描结果里有误报吗？"
**A**: 有可能。ODC 引擎的 CPE 匹配基于组件名和版本号的模糊匹配，不是 100% 精确。DC 原生支持 suppression 文件来屏蔽已知误报——我在 `application.yml` 里预留了 suppression 配置项，但还没在 UI 里做手动标记的功能。这是后续迭代点。

## 三、DC 引擎类

### Q: "DC 9.0.0 的作者是谁？为什么选这个版本？"
**A**: Jeremy Long，OWASP 官方项目 maintainer。选 9.0.0 是因为 5 月底启动项目时它是稳定最新版。12.x 当时刚起步，API 变化大、文档不全。67 个测试跑在 9.0.0 上——升级有规划但安排在下一期。

### Q: "引擎自己能扫多少种文件格式？"
**A**: Jar Analyzer（.jar 内的 MANIFEST + pom.properties）、Archive Analyzer（解压递归扫描）、POM Analyzer（Maven pom.xml）、NPM/Node Audit、Python、Ruby Gem、.NET Assembly、Autoconf 等 20+ 种。我们演示的三个项目覆盖了 Java/Jar 分析——但引擎能力远不止这三个。

### Q: "NVD 数据和 OSS Index 有什么区别？"
**A**: NVD 是 NIST 维护的国家漏洞数据库（权威、免费、覆盖面广但可能有延迟）。OSS Index 是 Sonatype 的商业数据（更快但需要账号、有速率限制）。当前系统 OSS Index 默认关闭——没有商业账号的前提下硬开会导致 401 扫描失败。这也是我加的配置项之一。

## 四、前端类

### Q: "为什么用 Element Plus？"
**A**: Vue 3 生态最成熟的企业级 UI 库，表格、分页、进度条、消息提示都是内置组件，不需要从零造轮子。项目定位是工具平台而非设计展示，开发效率优先。

### Q: "前端轮询方案有什么缺点？"
**A**: 3 秒轮询在任务数量少时没问题，但也存在三个局限：一是轮询间隔内状态变化有延迟；二是大量并发任务时轮询请求会堆积；三是移动端耗电。WebSocket 推送会是更好的长期方案——我选了轮询是因为实现时间短、满足演示和少量用户的场景，且不引入额外依赖。

## 五、工程素养类

### Q: "67 个测试覆盖了哪些？"
**A**: 9 个测试类：OssService（条件装配 + 业务逻辑）、ProjectService（ZI P 解压 + 创建/查询）、ScanTaskService（状态机流转）、ScanTaskExecutorService（线程池 + Engine 集成）、ReportService（报告路径/生成）等。覆盖 Controller 到 Service 到 Mapper 的全链路，使用 Mockito + JUnit 5。

### Q: "199 项核查清单是什么？"
**A**: 一份自己写的结构性自查清单——每完成一个模块就逐项打勾。覆盖 A 类（核心逻辑正确性）、B 类（边界情况与性能）、C 类（异常处理与安全脱敏）、D 类（配置与部署）、E 类（代码质量）、F 类（文档完整性）、G 类（深度审查）。46 个问题包含 5 个严重（如仪表盘 totalProjects 为 0、NVD 下载失败）全部修复。

### Q: "如果你再做一个类似项目，会怎么做不同？"
**A**: 三点——第一，先写测试再写代码（当前是写完再补测试，导致 Service 层测试覆盖率稍低）。第二，文档在开发过程中持续更新而非最后集中补（7/14 补了 4 份文档）。第三，一开始就用 Flyway 管理数据库迁移而非依赖 Hibernate DDL auto——现在迁移到 MySQL 时要手动写一次建表语句。

### Q: "你觉得这个项目实际能用吗？"
**A**: 目前能作为团队内部的依赖安全自查工具——一个人管理几十个项目的扫描，功能完整。但要作为企业级平台，还需补齐三点：认证鉴权、MySQL 迁移（H2 不适配多用户写入）、CI/CD 流水线集成。这些都在技术规划文档里。

## 六、现场不会让你挂的问题（如果被问到直接这样说）

### Q: "OAuth 2.0 / SSO / LDAP 认证怎么做？"
**A**: Spring Security + OAuth2 Client 可以在不修改业务 Controller 的前提下加上认证。我预留了扩展点——当前 Security 没引入是为了聚焦核心扫描功能的研发。

### Q: "你的报告里的漏洞评分是怎么来的？"
**A**: CVSS 评分由 NVD 官方提供，不是我算的。我的系统只是展示和分类——CRITICAL（≥9.0）、HIGH（7.0-8.9）、MEDIUM（4.0-6.9）、LOW（<4.0）。

### Q: "DC 引擎 9.0.0 到 12.x 有什么 breaking change？"
**A**: Settings 类重构——9.0.0 用 `new Settings()` 手动设键值对，12.x 改用 Builder 模式 + 键名变更。升级需要重写 `ScanEngineService` 的配置段，但 `Engine` 核心 API（`scan()` + `analyzeDependencies()`）基本兼容。
